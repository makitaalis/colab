# Документация контроль

## Проверка структуры docs

```bash
find Docs/Проект -maxdepth 2 -type f | sort
```

## Проверка ключевых ссылок

```bash
rg -n "Документация \(архитектура и правила\)|Шаблон документа модуля|Промпты Codex \(документация\)" Docs/Проект/INDEX.md README.md
```

## Проверка ссылок в INDEX

```bash
tmp=$(mktemp)
rg -o '`[^`]*\/[^`]*`' Docs/Проект/INDEX.md | tr -d '`' > "$tmp"
while IFS= read -r p; do [ -z "$p" ] && continue; [ -e "$p" ] || echo "MISSING: $p"; done < "$tmp"
rm -f "$tmp"
```

## Стандарты

- `Docs/Проект/Документация (архитектура и правила).md`
- `skills/orangepi-passengers-doc-architecture/SKILL.md`

## Обязательная фиксация после каждого этапа

После каждого изменения кода/конфига этап считается завершённым только если обновлены:

1. профильный модуль (`Docs/Проект/Модули/...`);
2. операционный runbook (`Docs/Проект/Операции/...`);
3. troubleshooting при необходимости (`Docs/Проект/Проблемы/...`);
4. `Docs/Проект/Паспорт релиза документации.md`.

Минимальный формат записи в паспорт:

- что сделано;
- зачем сделано;
- результат/проверка;
- ошибки/ограничения (если были);
- следующий этап и зачем.
